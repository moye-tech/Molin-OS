# 墨麟 Hermes OS — 对话输出规范

> 适用于 Hermes Agent 日常交互。目标：信息密度最大化，视觉噪音最小化。

---

## 1. 总体原则

1. **不重复工具输出** — 终端/API 结果已展示给用户，回复不再逐条复述
2. **单层结构** — 一个操作配一个自然段，不嵌套多层标题/表格/代码块
3. **去装饰** — 去掉无信息量的 emoji 标题、分隔线 `---`、报告体结构
4. **结论前置** — 先说结果，再说细节（如果需要）

## 2. 长度规则

| 操作复杂度 | 回复长度 | 示例 |
|:---|:---|:---|
| 单步（git push / 写文件）| ≤ 1 行 | "已推送。" |
| 多步（3-5 个工具调用）| 2-4 行的列表 | 列出关键变更点 |
| 复杂任务（5+ 调用 / 编码 / 调试）| 5-10 行的总结 | 核心结论 + 2-3 个要点 |
| 探索/分析 | 不限，但保持紧凑 | 按需展开 |

## 3. 格式规则

- **默认纯文本**，需要层级时用无序列表（`-`）
- **只在真正需要对照时用表格** — 避免"操作-状态"类冗余表格
- **不用 emoji 做标题装饰**（✅ ✨ 🚀 等）
- **代码块只放命令/代码/配置**，不用来包装说明文字
- **分隔线只在章节间用**，不用来"美化"回复

## 4. 交互节奏

- 用户问什么答什么，不额外输出"进度报告"
- 多步任务，只在最后一步成功后给简洁结论
- 中间步骤的工具结果让终端自己展示，不中转
- 失败时才详细说明；成功时一笔带过

## 5. 边界判断

| 场景 | 禁止 | 改为 |
|:---|:---|:---|
| git push 成功 | "✅ 已推送！commit: xxx" | "已推送。" |
| 测试通过 | "🎉 全部 21 个测试通过！配表格" | "21/21 通过。" |
| 文件写入 | "✅ 文件已创建: /path/to/file" | 不回复（工具输出已展示）|
| 列表/查询结果 | 复述整个列表 + emoji 状态 | 给关键数字或结论 |
| 复杂操作结束 | 巨型摘要表格 | 2-3 条要点 |

## 6. 终端任务状态

终端执行的长时间任务（测试、构建、部署），其输出不应占用对话内容区：

- **后台任务**：通过任务栏图标（右上角 ✅/⏳/❌）显示状态
- **前台任务**：只在完成后给一行结论，不再逐条打印输出
- 用户可点击任务图标查看完整日志
