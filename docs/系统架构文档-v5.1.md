## 2. CEO 推理引擎（核心大脑）

### 2.1 设计理念

废弃所有关键词匹配/语义缓存/硬编码路由，改为 **纯 LLM 推理的多轮对话引擎**。

CEO 不"猜"用户要什么，而是通过对话**推理出**用户要什么。

### 2.2 推理流程

```
用户: "帮我做个东西"
    │
    ▼ LLM推理（DeepSeek Flash, ¥0.00014）
    │
    ├─ 判断：信息不足 → clarify
    │    └── CEO: "你想要文案、设计、视频、客服方案、数据分析...哪种？"
    │         用户: "写一篇小红书文案推广AI绘画"
    │              │
    │              ▼ LLM再次推理（结合对话历史）
    │              │
    │              └── 判断：信息充足 → dispatch
    │                   ├── 子公司: ["墨笔文创"]
    │                   └── 交付物规格: "一篇推广AI绘画的小红书文案，
    │                       包含标题、正文（突出便捷性/创意性/场景）、
    │                       标签（#AI绘画 #人工智能），约500字"
    │
    ├─ 判断：纯闲聊 → chat
    │    └── 友好回复，零子公司调度
    │
    ├─ 判断：一次性明确请求 → dispatch
    │    └── 如"开发一个Python自动化脚本+Web界面" → 直接dispatch
    │
    └─ 判断：非法/超范围 → refuse
         └── 明确拒绝，不执行
```

### 2.3 核心技术

| 组件 | 路径 | 行数 | 职责 |
|:----|:----|:---:|:-----|
| `CEO` | `molib/ceo/ceo.py` | 76 | 决策入口，管理对话会话 |
| `CEOReasoningSession` | `molib/ceo/ceo_reasoning.py` | 373 | 推理引擎：上下文构建 + LLM调用 + 结果解析 |
| `SUBSIDIARY_CAPABILITIES` | `ceo_reasoning.py` | 20条 | 20家子公司能力画像，LLM推理依据 |

### 2.4 关键特征

- **推理模型**: DeepSeek V4 Flash (¥0.00014/次)，路由决策不需要 Pro
- **会话持久化**: JSON 文件 `~/.hermes/ceo_sessions/<id>.json`
- **上下文窗口**: 最近3轮对话（6条消息）
- **推理引导词**: 3种轮换模板（避免 LLM 模式固化）
- **输出**: `CEOAction{ action_type, message, subsidiaries, deliverable_spec, confidence }`

## 4. 两阶段执行与质量门控

### 4.1 Phase 1: 工具执行

| 组件 | 文件 | 行数 |
|:----|:----|:---:|
| `Phase1Executor` | `molib/ceo/phase_executor.py` | 536 |

`Phase1Executor.plan()` 基于 `deliverable_spec` 自动规划工具集：

```python
tool_plan = {
    "tools": [
        {"name": "web_search", "params": {"query": "AI绘画竞品分析", "count": 5}},
        {"name": "data_collect", "params": {"source": "行业报告", "dimensions": "市场规模/增长率"}},
    ],
    "gather_instructions": "搜集AI绘画工具的市场数据和竞品信息"
}
```

5种内置工具模板：
| 工具 | 用途 | 适用场景 |
|:----|:----|:---------|
| `web_search` | 网页搜索 | 竞品调研、市场分析、技术情报 |
| `data_collect` | 数据采集 | 财报分析、用户数据统计 |
| `file_operation` | 文件操作 | 读取已有文档、保存输出 |
| `code_execution` | 代码执行 | 数据清洗、格式转换 |
| `api_query` | 外部API | 查评分、获取天气、查汇率 |

### 4.2 Phase 2: LLM 合成

`Phase2Executor.synthesize()` 基于原始素材 + 子公司质量标准进行合成。

### 4.3 质量标准（按子公司）

| 子公司 | 通过线 | 5条评分标准 |
|:------|:-----:|:-----------|
| **墨笔文创** | 6/10 | 原创性、受众匹配、结构完整、可执行性、语言质量 |
| **墨图设计** | 6/10 | 创意性、一致性、可用性、清晰度、可行性 |
| **墨码开发** | 7/10 | 功能完整性、代码质量、可维护性、正确性、文档完整 |
| **default** | 6/10 | 完整性、准确性、可用性、结构化、价值 |

### 4.4 质量门控升级链路

```
Flash (¥0.00014) → 评分 < 6 ?
    ↓ 是
Pro (¥0.0004)   → 评分 < 6 ?
    ↓ 是
Reasoner (¥0.00055) → 评分 < 6 ?
    ↓ 是
返回最佳版本 + "未达标"标注
```

升级路径**单向递进**，不会被降级使用。

## 7. 输出格式化与噪声过滤

### 7.1 功能

| 特性 | 说明 |
|:----|:------|
| **噪声过滤** | 自动移除 terminal()、read_file()、临时路径、中间步骤等所有工具调用痕迹 |
| **结构化卡片** | 分隔线 + 标题 + 键值字段 + 要点 + 操作提示 |
| **分级透明** | L0 自动完成 / L2 等待审批 / L3 董事会决策 |
| **错误卡片** | ❌ 统一格式，含错误原因和操作指引 |

### 7.2 噪声过滤规则

| 噪声类型 | 处理 | 示例被过滤 |
|:---------|:----|:-----------|
| terminal命令 | 完全移除 | `terminal("cd /tmp/...")` |
| 文件操作 | 完全移除 | `read_file("/path/...")` |
| 中间验证 | 完全移除 | `来验证一下...` |
| 临时路径 | 完全移除 | `/tmp/`, `~/.cache/` |
| 技术元数据 | 完全移除 | `iteration 12/60` |
| 冗余前言 | 精简 | `让我先检查一下` → 移除 |

### 7.3 卡片模板

**任务完成卡片：**
```
━━━━━━━━━━━━━━━━━━━━
  ✅ [操作名]
━━━━━━━━━━━━━━━━━━━━

· 字段1：值
· 字段2：值

· 要点
· 要点

━━━━━━━━━━━━━━━━━━━━
```

**审批卡片：**
```
━━━━━━━━━━━━━━━━━━━━
```
